# Writing style guide

Follow these style guidelines for all English text produced or modified in this project.

**Punctuation and formatting**

* Use straight quotes (`"`, `'`) instead of curly quotes.
* Use the Oxford comma (e.g., "red, green, and blue").
* Use `-` instead of en/em dashes.
* Use `->` instead of arrows.
* Single space after periods (unless Markdown line break).
* Do not split a sentence across multiple lines. Keep each sentence on a single line.
* No trailing punctuation in headings and labels.
* Keep numerals as numerals. Do not convert numerals (1, 2, 3, etc.) into words unless explicitly instructed. This applies to all text generated or modified: code comments, docs, changelogs, release notes, commit/PR messages, console output summaries, test plans, and prompts embedded in repo docs.

**Tone and wording**

* Avoid contractions (write "do not" not "don't").
* No slang, idioms, or culturally specific expressions.
* Keep wording simple and direct for non-native English speakers.
* Use active voice.
* Address the user directly with "you."
* Be concise - cut filler like "please note that", "in order to", "basically."
* Use positive framing - tell users what they can do, not what they cannot.
* Do not anthropomorphize the system.

**Capitalization**

* Sentence case for headings, subheadings, buttons, and labels.
* Consistent capitalization and punctuation across similar elements.
* No ALL CAPS for emphasis - use bold or italics.

**Clarity**

* Lead with the objective or outcome, not the action.
* Use parallel structure in lists.
* No double negatives.
* Specific, descriptive labels for buttons and links.
* Error messages should explain what happened and what to do next.

**Why:** User explicitly requested this style guide be followed for all writing in the project.

**How to apply:** Apply these rules to all English text: README files, docs, UI copy, commit messages, PR descriptions, and any other user-facing text.
